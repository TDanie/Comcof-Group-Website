# Comcof Digital Commerce: Architecture

## Existing architecture (audited 2026-07-19)

- Marketing site: static HTML (18 routes), shared `assets/style.css` and
  `assets/site.js`, deployed on Vercel (project `comcof-group-website`)
  with GitHub auto-deploy from `main`. Careers pages generated by
  `tools/build-jobs.js`. All routes verified live and preserved.
- No pre-existing backend, forms are structured mailto, no analytics.

## Decision: hybrid architecture

The working marketing site is not migrated. The shop is a separate
Next.js 15 (App Router, TypeScript, strict) application in `shop-app/`,
deployed as its own Vercel project (`comcof-shop`), and mounted on the
main domain through rewrites in the static site's `vercel.json`:

- `comcofgroup.com/shop/*`, `/account/*`, `/admin/*`, `/_next/*`
  proxy to `comcof-shop.vercel.app`.
- `shop.comcofgroup.com` can later point directly at the `comcof-shop`
  project; the commerce backend needs no rebuild for that move.

Rationale: zero risk to the live, indexed marketing site; the commerce
platform gets a real application framework; both share one repository
and one brand system (tokens ported into `shop-app/src/app/globals.css`).

## Modes

The shop runs in one of two explicit modes (see `src/lib/env.ts`):

- **LAUNCH** (no Supabase env): honest launch states, no invented
  products or prices, ordering via structured email, corporate enquiry
  form live. This is the current production state.
- **COMMERCE** (Supabase configured): database-backed catalogue,
  variants, accounts, carts, orders, admin. Activated by setting the
  environment variables in `.env.example` on the `comcof-shop` Vercel
  project and applying `supabase/migrations`.

## Backend

Supabase (PostgreSQL + Auth + RLS). Schema in
`supabase/migrations/0001_commerce_core.sql`: catalogue, variants,
inventory with movement history, orders with status history and
human-readable order numbers, payments with idempotent event log,
delivery zones, promotions, subscriptions, reviews (moderated),
corporate enquiries, support, loyalty (flagged off), settings, feature
flags, audit logs. Money is BIGINT whole UGX; currency stored
explicitly; RLS deny-by-default with public read limited to active
catalogue data.

## Payments (boundary defined, not yet integrated)

A provider adapter will live in `shop-app/src/lib/payments/` with
implementations added per provider (mobile money aggregator such as
Flutterwave or Pesapal for MTN/Airtel + cards, plus manual verification
and pay-on-delivery). Webhooks land in route handlers with signature
verification and idempotency via `payment_events`. No provider is
integrated until credentials exist; production never simulates payment.

## Phase status

- Phase 1 (audit, foundation, schema, brand system): complete.
- Phase 2 (catalogue, product pages, cart, filters): code complete,
  activates in COMMERCE mode; accounts/checkout server actions pending.
- Phases 3 to 6 (payments, admin, mobile app, tests at depth): pending;
  blocked primarily on Supabase provisioning and payment credentials.
